## 一、游戏概述

### 1.1 核心体验陈述
在这个游戏里，玩家感受到**在信息不完全的压力下管理风险的紧张感**——你不知道自己全部底牌，只能通过有限的记忆和观察来推断，在"再贪一轮"和"见好就收"之间做出抉择，并在自己觉得血糖最低的瞬间喊出"稳态"。

### 1.2 基本信息
| 项目 | 内容 |
|------|------|
| 游戏名称 | 血糖卡波（暂定） |
| 开发团队 | 个人独立开发 |
| 游戏类型 | 多人休闲卡牌游戏（Kabo 桌游改编） |
| 美术风格 | 休闲娱乐，可爱风 |
| 目标平台 | PC（Windows 优先） |
| 游戏引擎 | 客户端：Unity 2022.3.62f3c1（C#）；服务端：C++ / muduo / protobuf |
| 目标玩家画像 | 泛用户/大众休闲；18-34 岁为主；核心动机：社交互动 + 放松休闲 |
| 内容分级/边界 | 全年龄可用；医疗主题下避免血腥、恐怖或过度紧张的视觉元素 |
| 商业模式 | 免费游玩，不做商业化（仅供朋友间联机验证） |
| 游戏规模 | 小型项目（单人可交付） |
| 开发周期 | 预计 10 周完成可联机原型，18 周内完成完整 MVP（按里程碑） |
| 核心特色 | 赛博医疗主题下的 Kabo 暗牌博弈；信息不完全 + 心理博弈；极简操作深度策略 |

### 1.3 游戏简介
《血糖卡波》是一款多人休闲卡牌对战游戏，基于经典桌游 Kabo 改编，以糖尿病血糖控制为叙事包装。每局 2-6 名玩家，每人持有 4 张暗牌代表当前"代谢负荷"（血糖值）——你只能看到其中 2 张，另外 2 张全程盲。通过抽牌、替换、偷看、交换等操作控制自己的血糖总值，在自信最低时主动喊出"稳态"结束回合。多轮累计，总分最低者获胜。

设计上不做商业化，聚焦核心社交体验：易上手（每回合三选一）、高互动（偷看、换牌、心理战）、赛博医疗视觉包装。

### 1.4 目标玩家画像
| 维度 | 结论 | 对设计的影响 |
|------|------|--------------|
| 受众取向 | 泛用户 / 大众休闲 | 题材不偏向特定性别，用普适的医疗健康隐喻包装 |
| 年龄层 | 18-34 岁为主，不排除其他年龄段 | 规则可在 5 分钟内学会；无暴力/血腥内容 |
| 核心动机 | 社交互动 + 放松休闲 | 强调心理博弈和玩家互动，不做竞技排位；单局 10-15 分钟，适合碎片时间 |
| 游玩频率 | 碎片化短对局，朋友约局为主 | 不支持排位/天梯；匹配功能非 MVP 范围；好友房间为核心入口 |
| 内容边界 | 全年龄；避免恐怖元素 | 赛博医疗风保持"设备面板""读数""传感器"美学，不做手术/血液/针头等具象医疗视觉 |

### 1.5 参考游戏
| 参考游戏 | 借鉴点 | 差异化 |
|----------|--------|--------|
| **Kabo（桌游）** | 核心规则：4 暗牌、抽换偷看交换、喊 Cabo 结束、累计计分 | 数字化为在线多人；用血糖控制主题替换抽象数字；增加赛博医疗 UI 包装 |
| **Coup / Love Letter** | 轻量社交推理卡牌的设计哲学：少规则、多互动 | 不复制角色/能力体系，只借鉴"小体量高互动"的产品定位 |

### 1.6 市场竞品分析
| 竞品 | 可借鉴点 | 我们的差异化 | 需要避免的问题 |
|------|----------|--------------|----------------|
| Tabletop Simulator 上的 Kabo Mod | 已验证的 Kabo 数字玩法 | 独立客户端、专用 UI、血糖主题沉浸感更强；不需要 TTS 本体 | 仅靠 Kabo 规则难以长期留住玩家，需靠主题和社交体验弥补 |
| UNO（数字版） | 简单规则 + 多人社交的爆款模型 | 暗牌信息不完全带来的心理博弈深度远超 UNO 的纯运气；血糖主题更新颖 | UNO 有 IP 认知优势；我们需要靠独特主题突围 |
| Among Us | 信息不完全社交博弈的标杆 | 我们的博弈是数字/概率维度，而非狼人杀式的身份推理，更偏策略计算 | Among Us 依赖大规模玩家和语音；我们锁定 2-6 人小局 |

## 二、核心玩法设计

### 2.1 核心循环

```
┌─────────────────────────────────────────────────────────────┐
│                    一局游戏的核心循环                          │
│                                                             │
│  轮到你的回合                                                │
│     │                                                       │
│     ├── 选项A：从抽牌堆暗摸一张                                │
│     │     ├── 直接弃掉 → 回合结束                              │
│     │     ├── 替换自己一张暗牌 → 旧牌弃掉 → 回合结束             │
│     │     └── 使用技能（仅 7-Q）→ 发动 → 牌弃掉 → 回合结束       │
│     │                                                       │
│     ├── 选项B：从弃牌堆拿最上面一张（明牌）                      │
│     │     └── 替换自己一张暗牌 → 旧牌弃掉 → 回合结束             │
│     │                                                       │
│     └── 选项C：直接喊"稳态！"                                 │
│           └── 其他人各再走最后一回合 → 全员亮牌 → 计分 → 新轮开始  │
│                                                             │
│  多轮累计 → 有人总分 ≥100 → 游戏结束 → 总分最低者获胜          │
└─────────────────────────────────────────────────────────────┘
```

**核心决策张力：** 每一回合都是"获取更多信息/优化手牌"与"对手可能比你先喊稳态"之间的权衡。你越贪，被对手抢先的风险越大；你越早喊，自己手牌可能还不够低。

### 2.2 玩法深度

**上手门槛（低）：**
- 每回合三选一（暗摸 / 明拿 / 喊稳态）
- 数字比大小，不需要复杂计算
- 技能只有 4 种（偷看自己 / 偷看对手 / 换牌），效果直观

**策略深度（中）：**
- 记忆博弈：记住自己看过的 2 张 + 通过偷看获得的信息
- 概率推算：弃牌堆信息 = 缩小了的未知范围，可推断对手手牌分布
- 心理博弈：何时喊稳态？对手的表情/行为暗示（数字版通过操作节奏判断）
- 风险控制：K（看+换）可能改善也可能更差；J/Q 盲交换是双刃剑

**长期目标：**
- 单局内：本轮血糖最低
- 跨轮：累计总分最低
- 社交层面：与朋友建立"默契牌局"的娱乐记忆

### 2.3 主要系统

| 系统 | 功能 | 如何服务核心体验 |
|------|------|-----------------|
| 牌局系统 | 发牌、回合管理、抽牌堆/弃牌堆状态机 | 承载核心循环的基础设施 |
| 技能系统 | 7/8 偷看、9/10 间谍、J/Q 交换 | 提供信息获取与风险操作的策略维度 |
| 计分系统 | 亮牌、算分、稳态判定、累计追踪、100 分惩罚/奖励 | 将"信息不完全"的紧张感转化为可量化的胜负 |
| 房间系统 | 创建/加入房间、准备、开始 | 承载社交入口，是核心动机的实现载体 |
| 网络同步 | 回合状态、牌堆、玩家操作的可靠同步 | 保证多人体验的一致性 |

### 2.4 操作方式

| 平台 | 操作方式 |
|------|----------|
| PC（鼠标） | 点击自己的暗牌选择替换目标；点击牌堆执行抽牌；技能按钮自动弹出条件触发；右键查看已见过的牌备忘 |

**交互原则：**
- 每回合可执行操作不超过 5 个可点击区域（抽牌堆、弃牌堆、4 张暗牌），保持界面清爽
- 技能触发时自动弹出确认，不打断流畅感
- 各玩家的暗牌在自己视角中部分可见（左边 2 张已记住的），增强信息管理体验

## 三、系统详细设计

### 3.1 牌具系统

**专用血糖牌组（52 张）：**

| 血糖值 | 张数 | 点数 | 技能 | 主题含义 |
|--------|------|------|------|----------|
| 0 | 2 | 0 分 | 无 | 理想血糖——最完美状态 |
| 1–6 | 各 4 张（共 24 张） | 1–6 分 | 无 | 正常范围——安全但可优化 |
| 7–8 | 各 4 张（共 8 张） | 7–8 分 | **偷看**：看自己任意 1 张暗牌 | 血糖偏高——开始需要"监测" |
| 9–10 | 各 4 张（共 8 张） | 9–10 分 | **间谍**：看对手任意 1 张暗牌 | 血糖危险——需要"监测他人"作为参考 |
| 11–12 | 各 4 张（共 8 张） | 11–12 分 | **盲交换**：你选自己 1 张，对手选 1 张，互换（双方不能看） | 血糖失控——只能"交换代谢数据"碰运气 |
| 13 | 2 | 13 分 | **看+换**：先看对手 1 张，再决定是否与自己 1 张互换 | 极端高血糖——全面"诊断+干预" |

**设计要点：**
- 点数越高，技能越强，但留在手里代价也越大——天然风险/收益平衡
- 7-8（偷看自己）和 9-10（偷看对手）的信息获取方向不同，形成策略差异
- 11-12（盲交换）是高风险操作：可能把高牌换走，也可能换回更糟的
- 13（看+换）是最强单卡：先看再决定，可控性最高，但本身 13 分是最高负担

### 3.2 回合流程系统

**状态机：**

```
等待开始 → 洗牌发牌 → [玩家回合循环]
                            │
                    当前玩家回合开始
                            │
                  选择行动（A/B/C）
                     /      |      \
                    A       B       C
                  摸暗牌   拿明牌   喊稳态
                  /|\       |        |
             弃掉/替换/技能  替换     最终轮
                            |        |
                         回合结束  全员亮牌→计分→新轮/游戏结束
```

**回合规则细节：**
- 选项 A（暗摸）：摸到的牌可以立刻使用技能，如果它是 7-12 号牌
- 选项 B（明拿）：只能替换，不能发动技能——这是代价
- 选项 C（喊稳态）：喊完后其他人各走最后一回合，**这些人不能再次喊稳态**
- 技能使用后，那张技能牌进弃牌堆（不是留在手里）

### 3.3 计分系统

**单轮计分：**

| 情况 | 计分 |
|------|------|
| 喊稳态者，总分最小（或并列最小） | 本轮 0 分（"完美稳态达成"） |
| 喊稳态者，总分不是最小 | 自己 4 张总和 + **10 分罚分**（"稳态误判——代谢危机"） |
| 其他玩家 | 直接累计自己 4 张点数 |
| 特殊牌型（神风特攻队）：某玩家面前恰好是两张12和两张13 | 该玩家本轮得 0 分，其他所有玩家本轮各得 50 分 |

**全局计分：**
- 多轮累计，任一玩家总分 **≥100 分**，游戏立即结束
- 最终总分最低者获胜（若平分，则最后一轮得分最低者胜出）

**百分重置规则（翻大浪）：**
- 如果某位玩家的累计分数恰好达到 100 分，可以选择将分数重置为 50 分
- 整局游戏中每位玩家限用一次此机制
- 使用此机制后，游戏继续进行

### 3.4 房间与联机系统

**房间流程：**
```
创建房间 → 生成房间码 → 分享给朋友 → 加入 → 准备 → 开始游戏
```

**连接与房间流程：**
1) 客户端通过 HTTP/HTTPS 访问大厅（版本、房间列表、创建/加入）
2) 加入房间后升级为对局连接（WebSocket/WSS）
3) 服务端分配座位并下发房间状态（玩家列表、准备状态）
4) 全员准备 → 服务端权威发牌 → 广播对局快照

**房间功能（MVP）：**
- 创建/加入房间（通过房间码）
- 玩家列表与准备状态
- 2-4 人可开始（MVP），5-6 人后续扩展
- 掉线重连（60 秒内恢复，保留手牌与回合）

**同步策略：**
- 服务端权威：洗牌/抽牌/弃牌/技能结果只由服务端生成
- 客户端只发送操作意图，服务端校验后广播权威结果
- 关键操作后下发全量状态，保证一致性

**断线处理策略（MVP）：**
- 断线 60 秒内可带 `session_token` 重连
- 超过 60 秒进入托管（默认自动结束回合）
- 超过 120 秒仍未回：判定离局（房主可配置）

**不做（MVP 范围外）：**
- 随机匹配
- 大厅/排行榜
- 好友系统（直接分享房间码即可）
- 观战

## 四、数值设计框架

### 4.1 核心数值体系

**牌组数值：** 基于 Kabo 标准 52 张牌组，数值平衡已经过桌游社区验证。

| 统计项 | 数值 |
|--------|------|
| 牌组总数 | 52 张 |
| 均值 | 约 7.0 |
| 4 张暗牌的期望总和 | 约 28 分 |
| 单轮最低可能 | 2 分（0+0+1+1） |
| 单轮最高可能 | 52 分（4 张 13） |
| 稳态罚分 | +10 分 |

**平衡分析：**
- 罚分 +10 的设计：如果喊稳态时不是最低，代价约等于一张中等偏高的牌（10 分），足够痛但不会直接崩盘
- 游戏结束线 100 分：按均值 28/轮计算，约 3-4 轮不会出局；如果有意压低，可持续 5-8 轮
- 恰好 100 分的减 50 机制：提供戏剧性翻盘可能，鼓励玩家在接近 100 时更激进

### 4.2 游戏节奏

| 阶段 | 特征 | 策略重点 |
|------|------|----------|
| 开局（第 1-2 轮） | 所有人信息最少，保守试探 | 稳定替换，优先记住自己的牌 |
| 中局（第 3-5 轮） | 弃牌堆信息积累，部分玩家接近 100 | 技能性价比提升（已知信息更多），开始心理博弈 |
| 终局（6 轮+） | 多人接近 100 分，紧张感最高 | 激进或保守的抉择成为决胜负关键；100 分减 50 触发后的追赶 |

### 4.3 核心指标定义

> 个人独立项目，不设商业化指标。以下指标用于验证玩法质量和社交体验。

| 指标 | 目标值/观察方式 | 设计含义 |
|------|----------------|----------|
| 原型可玩率 | 朋友间 2-6 人能完成完整一局 | 核心循环技术可行 |
| 单局平均时长 | 10-15 分钟 | 碎片化时间的合适长度，不拖沓 |
| 稳态喊出率 | 每轮有 1 人以上喊稳态 | 核心机制被正常使用，不会无人敢喊 |
| 朋友重开意愿 | "再来一局"率 ≥ 70% | 社交乐趣和复玩价值 |
| 规则理解时间 | 新玩家 5 分钟内上手 | 休闲定位的学习成本 |
| 掉线重连成功率 | ≥ 95%（朋友测试阶段） | 网络可靠性 |
| 回合内操作延迟 | ≤ 500ms（跨网可接受） | 互动体验流畅度 |
| 状态一致性 | 回合广播后 1 秒内一致 | 规则可信度 |

**美术策略（个人项目）：**
- 所有 UI 元素优先使用几何图形 + 程序化渐变，避免手绘资产
- 卡牌设计用程序化模板生成，只需设计 1 套布局，数值动态填入
- 无需角色立绘、3D 模型或场景动画
- 可考虑使用免费赛博朋克 UI 素材包加速开发

## 六、技术架构建议

### 6.1 推荐引擎
- **客户端：Unity（C#）** — 已选，适合快速 UI 原型和跨平台扩展（如有需要）
- **服务端：C++ / muduo / protobuf** — 已选，高性能网络库适合回合制游戏的可靠同步

**推荐理由：** Unity 对 PC 桌面应用的 UI 开发成熟，C# 作为客户端语言与 protobuf 配合良好；muduo 的事件驱动模型适合处理多房间并发。栈选型整体匹配个人项目的效率需求。

### 6.2 核心技术点

| 技术点 | 方案 | 难度 | 备注 |
|--------|------|------|------|
| 网络通信协议 | WebSocket/WSS + protobuf | 中 | 兼容 Cloudflare Tunnel；客户端跨平台 |
| 房间管理 | 服务端房间状态机 | 中 | 房间创建/加入/离开/解散，掉线检测 |
| 回合状态同步 | 服务端权威 + 客户端不做预测 | 中 | 服务端为所有随机结果（洗牌、抽牌）的唯一权威源 |
| 内网穿透 | Cloudflare Tunnel（首选）/ frp / ZeroTier | 低 | 通过临时域名或虚拟局域网实现跨网联机 |
| 掉线重连 | session_token + 全量快照 | 中 | 重连时下发当前完整状态 |
| UI 框架 | Unity UI (uGUI) 或 UI Toolkit | 低 | 极简 UI，无需复杂框架 |

**服务端消息类型（protobuf 设计预览）：**
```
- RoomCreate / RoomJoin / RoomLeave
- GameStart / GameEnd
- PlayerTurnStart / PlayerAction / PlayerTurnEnd
- ActionDraw / ActionReplace / ActionSkill / ActionCallSteady
- RoundReveal / RoundScore / GameOver
- ReconnectRequest / ReconnectState
```

### 6.3 网络架构

```
┌──────────┐   ┌──────────┐   ┌──────────┐
│ 客户端 A  │   │ 客户端 B  │   │ 客户端 C  │
└────┬─────┘   └────┬─────┘   └────┬─────┘
     │              │              │
     └──────────────┼──────────────┘
                    │
            ┌───────┴───────┐
            │  游戏服务器     │
            │  (C++/muduo)  │
            │  - 房间管理     │
            │  - 回合状态机   │
            │  - 牌堆同步     │
            └───────────────┘
```

服务端为权威端，所有牌堆操作（洗牌、抽牌、弃牌）和随机结果由服务端决定并广播。客户端只做输入收集和表现。

### 6.4 网络协议与重连流程

**消息包结构（示例）：**
- `op`：操作码（Join/Action/StateSync/Chat）
- `seq`：客户端递增序号（用于幂等处理）
- `room_id` / `player_id` / `session_token`
- `payload`：具体动作或状态体（protobuf）

**重连流程：**
1) 客户端重连 → 发送 `ReconnectRequest(session_token)`
2) 服务端验证 → 返回 `StateSync`（全量状态）
3) 客户端覆盖本地状态 → 继续回合

## 七、MVP 与垂直切片

### 7.1 MVP 范围
| 模块 | 必须包含 | 暂缓/不做 | 验证目的 |
|------|----------|-----------|----------|
| 核心玩法 | 完整回合流程（抽牌/替换/技能/喊稳态）、计分（单轮+全局）、游戏结束判定 | 无 | 核心循环是游戏存在的理由 |
| 联机 | 创建/加入房间（房间码）、2-4 人联机、内网穿透方案 | 5-6 人、随机匹配、大厅 | 验证多人社交体验 |
| 牌具 | 标准 52 张血糖牌组，4 种技能 | 更多技能牌、特殊牌 | 验证信息不完全机制的乐趣 |
| 美术 | 极简 UI + 程序化卡牌模板 + 赛博配色 | 高级动效、粒子特效、适配多分辨率 | 验证主题包装完成度 |
| 技术 | 服务端权威回合同步、掉线重连 | 服务器性能优化、反作弊 | 验证基础可玩性 |
| 内容 | 单一游戏模式 | 教程、AI 对手、观战、数据统计 | 最小化内容量，聚焦核心 |

### 7.2 垂直切片
**目标场景：** A 创建房间 → 分享房间码给 B 和 C → 三人联机玩 **完整的一局**（多轮直到有人超过 100 分）→ 结算展示排名。

这个切片涵盖：
- 完整的房间创建到游戏结束闭环
- 所有核心操作（抽/换/技能/喊稳态）
- 赛博医疗 UI 的最终视觉方向
- 一局的完整情感弧线：开局谨慎 → 中期信息博弈 → 终局紧张冲刺

**可录制为 10-15 分钟演示视频，用于展示和收集反馈。**

### 7.3 核心假设
| 假设 | 验证方式 | 通过标准 |
|------|----------|----------|
| Kabo 桌游的数字体验同样有趣，手感不因缺实体牌而下降 | 朋友实机测试，观察操作流畅度和反馈 | 测试者表示"和玩实体差不多"或"更有趣" |
| 血糖赛博主题能提升沉浸感而非违和 | 测试后访谈 | 测试者能自然使用"血糖""稳态"等词汇而非"分数""卡波" |
| 内网穿透方案能满足朋友联机需求 | 跨网络测试 | 3 人以上不同网络环境下延迟 <500ms，无断连 |
| 极简 UI + 程序化卡牌足以支撑视觉品质 | 截图给他人评价 | 非开发者认为"看起来像完成的游戏"而非"开发者 Demo" |

## 八、开发里程碑

### Phase 1：单机核心原型（第 1-4 周）
**目标：** 在一个进程内跑通完整 Kabo 规则

| 交付物 | 说明 |
|--------|------|

**验证指标：** 自己能和家人/室友在热座模式下正常玩一局，没有规则 Bug。

### Phase 2：联网原型（第 5-10 周）

| 交付物 | 说明 |

| Unity 客户端联机 | 创建/加入房间、回合操作同步、状态更新 |
| 内网穿透方案 | Cloudflare Tunnel（首选）/ frp / ZeroTier 配置指南与验证 |

**验证指标：** 朋友在不同网络下能成功联机完成一局，掉线后能重连。

### Phase 3：主题包装与 UI 打磨（第 11-14 周）

| 交付物 | 说明 |

| UI 主题化 | 配色、字体、动效全部赛博医疗化 |
| 音效 | 基本的电子音效反馈 |
| 结算动画 | 亮牌 + 分数计算 + "稳态达成/误判"反馈 |

**验证指标：** 朋友评价"有内味了"（赛博医疗沉浸感确认）。

### Phase 4：测试与稳定（第 15-18 周）

| 交付物 | 说明 |

**验证指标：** 三人以上异地联机成功完成 3+ 局，无致命 Bug。

## 九、风险评估

### 9.1 技术风险
| 风险 | 影响 | 可能性 | 应对策略 |
|------|------|--------|----------|
| muduo + protobuf 学习曲线 | 开发进度延迟 | 中 | Phase 1 用 Unity 单机验证规则；网络层可先用简单 TCP + JSON 原型，确认可行后再迁移到正式栈 |
| Cloudflare 临时隧道不稳定或域名变化 | 朋友无法联机 | 中 | 客户端提供一键更新服务器地址；必要时切换为命名隧道或临时公网部署 |
| 回合同步 Bug | 牌堆状态不一致 | 中 | 服务端权威架构 + 每步操作后全量下发给所有客户端，不依赖增量同步 |
| 掉线处理复杂 | 一局被迫中断 | 低 | 初期容忍人工处理（掉线者判负），后续再完善重连 |

### 9.2 市场风险
| 风险 | 影响 | 可能性 | 应对策略 |
|------|------|--------|----------|
| Kabo 知名度低 | 无人理解游戏概念 | 中 | 不做商业化推广；目标用户就是自己和朋友；如需扩大，可在游戏内嵌入规则教学 |
| 血糖主题可能让部分人不适 | 受众缩小 | 低 | 全年龄向包装，赛博化而非具象化医疗场景，避免"疾病感" |
| 缺乏留存钩子 | 玩几局就腻 | 低 | 不做长期留存目标；社交卡牌的核心是"有人约就玩"，不是"每天签到" |

### 9.3 生产风险
| 风险 | 影响 | 可能性 | 应对策略 |
|------|------|--------|----------|
| 单人全栈开发效率瓶颈 | 进度延期 | 高 | 严格控制范围：不做匹配、不做 AI、不做排行榜、不做多语言。Phase 1 热座模式可独立验证玩法 |
| 美术产出不足 | UI 品质不够 | 中 | 程序化生成 + 极简风格本身就是设计方向，不依赖手绘资产 |
| 动机耗尽 | 项目搁浅 | 中 | 每个 Phase 都有可交付可玩版本，朋友反馈是持续动力来源 |
